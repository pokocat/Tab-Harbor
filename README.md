# TabHarbor 插件方案设计

## 1. 项目目标

开发一个 Chrome 插件，帮助用户在标签页过多时快速“看清、整理、收纳、恢复”标签页，降低以下常见问题：

- 打开了太多标签页，不知道哪些有用
- 同主题内容分散在多个窗口或多个时间段里
- 想暂时关闭一批标签页，但担心之后找不回来
- 标签页越积越多，浏览器变卡，注意力被打散

这个插件的核心不是“做一个新的书签工具”，而是成为用户管理“当前工作上下文”的中间层。

## 2. 目标用户

优先面向以下三类用户：

- 研究型用户：查资料时会连续打开大量文章、文档、视频、Issue
- 多任务用户：同时推进工作、学习、购物、娱乐等多个主题
- 拖延型囤标签用户：知道标签太多，但没有动力逐个清理

## 3. 产品定位

一句话定位：

“把失控的 tabs 变成可整理、可分组、可归档、可恢复的任务上下文。”

与现有书签/标签组的区别：

- 书签偏长期收藏，不适合高频临时整理
- Chrome 原生标签组偏展示和手动管理，缺少批量分析和归档能力
- 本插件强调“轻整理 + 快归档 + 易恢复”

## 4. 核心设计原则

- 低打扰：尽量不强迫用户重构自己的浏览习惯
- 快速见效：第一次使用就能明显减少标签混乱
- 可逆操作：任何清理动作都应该容易恢复
- 渐进增强：先解决 80% 问题，再逐步加入智能能力

## 5. 功能方案

### 5.1 MVP 核心功能

#### A. 标签总览面板

打开插件后，展示当前所有标签页的结构化列表。

建议信息维度：

- 页面标题
- 域名
- 所属窗口
- 最近访问时间
- 钉住状态
- 音频播放状态
- 重复标签标记

价值：

- 用户第一次就能看清“我到底开了什么”

#### B. 一键发现重复标签

检测同 URL 或近似 URL 的重复页面，例如：

- 完全相同 URL
- 仅 query 参数不同但主体相同的页面
- 同一文档/同一视频被打开多次

可提供操作：

- 保留最近使用的那个
- 一键关闭其余重复 tabs

价值：

- 这是最容易让用户立刻感受到价值的功能之一

#### C. 按主题快速分组

按域名或窗口对 tabs 做快速聚合，用户也可以手动创建“会话组”。

建议支持三种视图：

- 按窗口查看
- 按域名查看
- 按自定义分组查看

自定义分组示例：

- 招聘调研
- React 重构
- 周末旅行

价值：

- 帮用户从“一个个 tab”切换到“一个个任务上下文”

#### D. 一键归档一组标签

允许用户把当前选中的多个标签页保存成一个“会话”并关闭它们。

归档内容包括：

- 会话名称
- 保存时间
- 标签列表
- 来源窗口信息

后续支持：

- 一键恢复整组标签
- 在新窗口恢复
- 恢复到当前窗口

价值：

- 用户终于可以安心关闭一批标签，而不是一直“先留着”

#### E. 清理建议

插件基于简单规则给出“建议清理项”：

- 重复 tabs
- 长时间未激活 tabs
- 空白页 / 新标签页
- 已失效页面

建议交互：

- 用建议卡片展示可清理数量
- 用户确认后执行，不默认自动关闭

### 5.2 增强功能

#### F. 稍后再读 / 稍后处理

把标签页移动到“稍后处理”列表，而不是立即关掉并丢失上下文。

适合场景：

- 文章还没看
- 商品还没比较完
- 文档之后再整理

#### G. 智能命名会话

根据标签的标题、域名和关键词为会话自动生成名字，例如：

- “Chrome Extension Research”
- “电商比价”
- “前端性能优化资料”

#### H. 工作模式切换

用户可保存多个工作模式，一键切换：

- 工作
- 学习
- 娱乐

切换时可关闭当前上下文并恢复目标上下文。

#### I. 搜索全部打开与归档 tabs

支持搜索：

- 打开的标签
- 历史归档会话
- 某个域名下的全部页面

这个能力会显著增强恢复效率。

## 6. 推荐交互形态

### 6.1 插件入口

建议先做两层入口：

- 浏览器工具栏 popup：适合快速查看、清理、归档
- 独立管理页 `chrome-extension://.../dashboard.html`：适合深度整理大量标签

设计判断：

- Popup 适合轻操作，但空间很小
- 真正管理几十上百个 tabs 时，更需要完整页面

所以推荐：

- Popup 做“快捷入口”
- Dashboard 做“主战场”

### 6.2 Popup 结构

建议模块：

1. 顶部摘要
2. 清理建议
3. 当前窗口 tabs 预览
4. 快捷按钮

顶部摘要展示：

- 当前标签数
- 窗口数
- 重复标签数
- 可归档标签数

快捷按钮建议：

- 查看全部
- 关闭重复项
- 归档当前窗口
- 打开管理页

### 6.3 Dashboard 结构

建议三栏布局：

- 左侧：视图切换和筛选
- 中间：标签列表 / 分组列表
- 右侧：详情和批量操作

左侧筛选建议：

- 全部标签
- 当前窗口
- 重复项
- 长时间未使用
- 已归档会话
- 稍后处理

中间列表支持：

- 多选
- 拖拽分组
- 排序
- 按域名折叠

右侧操作建议：

- 关闭选中
- 归档选中
- 移入会话
- 移到新窗口
- 钉住 / 取消钉住

## 7. 典型用户流程

### 流程 1：快速清理

1. 用户点开插件
2. 看到“你有 18 个重复 tabs，9 个 7 天未访问 tabs”
3. 点击“查看建议”
4. 预览将被处理的 tabs
5. 确认后批量关闭或归档

### 流程 2：保存当前研究上下文

1. 用户正在查一个主题，已经打开 25 个页面
2. 想先切回别的工作
3. 点击“归档当前窗口”
4. 输入会话名或接受自动命名
5. 插件关闭这些 tabs，并保存会话
6. 以后可一键恢复

### 流程 3：找回之前看过的页面

1. 用户记得之前看过某篇文章，但不知道在哪个窗口
2. 在 Dashboard 搜索关键词或域名
3. 找到归档会话或当前打开页面
4. 一键激活或恢复

## 8. MVP 范围建议

建议第一版只做最能体现价值的功能，避免一开始就过度复杂。

### 第一版必须有

- 当前 tabs 总览
- 按窗口 / 域名查看
- 重复标签检测
- 批量选择与关闭
- 会话归档与恢复
- 本地搜索

### 第一版先不做

- AI 自动分类
- 跨设备同步
- 自动定时清理
- 复杂规则引擎
- 高级可视化图谱

这样可以更快验证用户是否真的愿意用“归档会话”和“清理建议”。

## 9. 技术实现方案

## 9.1 插件架构

建议基于 Chrome Extension Manifest V3。

模块划分：

- `popup`：快速入口
- `dashboard`：完整管理界面
- `background service worker`：管理 tab 数据、监听事件、执行批量操作
- `storage`：保存归档会话、用户偏好、规则配置

主要使用的 Chrome APIs：

- `chrome.tabs`
- `chrome.windows`
- `chrome.tabGroups`
- `chrome.storage.local`
- `chrome.runtime`
- `chrome.sessions`（可选，做恢复增强）

## 9.2 数据模型建议

### TabItem

- `id`
- `windowId`
- `title`
- `url`
- `favIconUrl`
- `pinned`
- `active`
- `audible`
- `lastAccessed`
- `groupId`

### SavedSession

- `id`
- `name`
- `createdAt`
- `updatedAt`
- `tabs`
- `sourceWindowId`
- `color`

### UserPreference

- `groupByDefault`
- `staleThresholdDays`
- `showDuplicateHints`
- `autoCollapseByDomain`

## 9.3 重复标签识别策略

建议先用“规则法”，不要一开始上 AI。

分三层：

1. 完全重复
   - URL 完全相同

2. 标准化后重复
   - 去掉 hash
   - 去掉常见追踪参数，如 `utm_*`
   - 对部分站点保留核心 query 参数

3. 近似重复
   - 同域名 + 相似标题
   - 同文档 ID / 视频 ID

MVP 先做前两层就够了，可控且解释性强。

## 9.4 存储策略

优先本地存储：

- `chrome.storage.local`

原因：

- 实现快
- 不依赖登录
- 用户对标签数据通常更敏感，本地优先更容易建立信任

后续可考虑：

- 导出 / 导入会话 JSON
- 云同步

## 10. 信息架构建议

插件中建议区分 4 类对象：

- 正在打开的 tabs
- 临时整理出来的分组
- 已归档的会话
- 稍后处理列表

这样用户心智会更清楚：

- “开着”是现在
- “归档”是以后可恢复
- “稍后处理”是短期待办

## 11. 视觉与交互风格建议

整体风格建议：

- 简洁、高信息密度，但不要像浏览器任务管理器那样生硬
- 重点突出“数量、风险、建议”
- 危险操作前提供二次确认或撤销

建议加入的细节：

- 域名颜色标签
- 重复项醒目标记
- 最近活跃标签页高亮
- 批量操作后 toast 提示和撤销入口

## 12. 风险点与设计取舍

### 风险 1：功能太多，用户反而更不会用

解决方式：

- 把首页限制为“看问题 + 做 1 到 2 个关键动作”
- 高级功能放进 dashboard

### 风险 2：误关标签造成不信任

解决方式：

- 所有批量关闭都先预览
- 提供撤销
- 优先归档再关闭

### 风险 3：重复检测不准

解决方式：

- 初期只对高置信规则做建议
- 不要静默自动操作

### 风险 4：性能问题

解决方式：

- 列表虚拟化
- 增量刷新
- 不在 UI 层做重计算

## 13. 版本路线图

### V1：验证价值

- Tabs 总览
- 重复检测
- 批量关闭
- 会话归档 / 恢复

目标：

- 验证用户是否愿意用插件完成“清理”和“上下文切换”

### V1.5：提升效率

- 稍后处理
- 搜索
- 会话自动命名
- 更好的筛选器

### V2：智能化

- 智能分类建议
- 根据行为推荐归档
- 跨设备同步

## 14. 成功指标

可以重点观察以下指标：

- 用户每周归档会话次数
- 用户每次清理减少的标签数
- 会话恢复率
- 重复标签清理使用率
- 7 日留存

如果“归档后又恢复”的行为很多，说明这个产品方向是成立的，因为它解决的是上下文管理问题，而不是单纯清理。

## 15. 我建议你先 review 的几个关键问题

为了更快收敛方案，建议你优先判断这几件事：

1. 产品主价值要偏“清理标签”，还是偏“保存/切换工作上下文”
2. 第一版入口是否接受“Popup + Dashboard”双界面方案
3. MVP 是否只保留“重复检测 + 会话归档 + 恢复”这三大核心能力
4. 是否希望产品偏工具效率风，还是稍微带一点智能助手感

## 16. 推荐结论

如果目标是尽快做出一个真正有机会被持续使用的 Chrome 插件，我推荐：

- 产品主线定为“会话归档 + 恢复 + 清理建议”
- 第一版聚焦高频刚需，不做复杂 AI
- 交互上采用“Popup 做快操作，Dashboard 做深管理”
- 技术上采用 Manifest V3 + 本地存储

这条路线实现成本可控，也最容易在早期拿到真实反馈。
